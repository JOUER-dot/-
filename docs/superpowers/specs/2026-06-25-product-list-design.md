# 组合产品管理页设计方案

日期：2026-06-25
目标页面：`frontend/src/views/advisor/ProductList.vue`
适用场景：金融投顾项目投顾端产品运营工作台

## 1. 设计目标

将当前“组合产品管理”页面从标准后台列表页升级为“任务处理优先”的投顾工作台。

页面需要优先解决以下问题：

- 首屏重点不清晰，任务处理和经营概览权重接近
- 筛选区分散，快捷筛选与详细筛选割裂
- 表格承载信息较多，但任务属性不够强
- 批量操作存在但状态感不足
- 页面整体更像通用后台，金融专业感不够突出

本次设计只聚焦前端页面结构、视觉层级和交互组织，不调整接口、查询参数、分页逻辑与业务动作本身。

## 2. 页面定位

页面定位为：

**投顾端产品运营工作台**

核心目标不是“浏览所有产品”，而是帮助投顾高效完成以下任务：

- 处理草稿
- 跟进待审核产品
- 处理驳回待改产品
- 管理已发布和可下架产品

因此，信息组织必须遵循：

**先任务，再待办，再筛选，再列表。**

## 3. 设计方向

最终采用的方向为：

**任务工作台型**

原因：

- 最符合投顾端的真实使用习惯
- 能把页面核心焦点从“数据罗列”转向“任务处理”
- 更容易建立金融专业中台的视觉气质

不采用纯列表管理型或双栏控制台型作为第一版，原因分别是：

- 纯列表管理型无法有效拉开与普通后台的差距
- 双栏控制台型实现成本更高，不适合作为本轮第一版

## 4. 页面结构

页面最终结构调整为：

1. 顶部工作区
2. 待处理任务区
3. 优先处理区
4. 经营概览区
5. 统一筛选工具带
6. 结果摘要区
7. 产品列表区
8. 批量操作区
9. 审核记录侧边区

阅读顺序为：

**先看当前最重要的任务，再看紧急项，再进入筛选和结果处理。**

## 5. 各区块设计

## 5.1 顶部工作区

职责：

- 告诉用户当前页面的核心任务
- 提供最重要的操作入口
- 汇总关键待处理状态

建议内容：

- 标题：组合产品管理
- 说明：管理草稿、跟进审核、维护已发布产品
- 摘要数字：
  - 待审核数量
  - 驳回待改数量
  - 近 7 天变更数量
- 主按钮：创建产品
- 次按钮：查看流程说明或导出列表

设计原则：

- 更像工作区总控面板，而不是传统大横幅
- 信息主次为：标题 > 摘要数字 > 主按钮 > 补充说明
- 允许轻度品牌蓝浅底，但避免大面积重渐变

## 5.2 待处理任务区

职责：

- 把待审核、驳回待改、草稿、可下架等高频任务从普通信息中抽离出来
- 一键驱动列表筛选

保留四张任务卡：

- 待处理草稿
- 待审核
- 驳回待改
- 可下架产品

每张任务卡包含：

- 任务名称
- 数量
- 一句动作说明

交互：

- 点击任务卡后立即更新 `queryForm.status`
- 当前激活卡有明显高亮
- 任务卡始终作为首屏第一优先级内容

颜色语义：

- 待审核：蓝色系
- 驳回待改：红色系
- 草稿：中性灰蓝
- 可下架：绿色系

## 5.3 优先处理区

职责：

- 展示当前最值得立即处理的 3 到 5 条产品
- 作为列表之前的紧急待办区

每条待办建议展示：

- 产品名称
- 产品类型
- 风险等级
- 最近更新时间
- 当前状态
- 一条任务原因说明
- 主操作按钮

规则：

- 驳回项优先级高于待审核项
- 按更新时间倒序排列
- 卡片样式应明显区别于表格行

视觉规则：

- 驳回项可使用浅红提示边
- 待审核项可使用浅蓝提示边
- 不做成通知列表，也不做成普通表格

## 5.4 经营概览区

职责：

- 补充展示整体运营数据
- 不参与首要任务判断

建议指标：

- 产品总数
- 已上架数量
- 待审核数量
- 近 7 天变更数量

设计原则：

- 视觉权重低于任务区
- 用于支持判断，不与任务卡竞争

## 5.5 统一筛选工具带

职责：

- 承接从任务卡进入后的细化过滤
- 将当前零散的筛选方式合并为一个统一工作区

结构：

第一层：

- 状态快捷筛选
- 类型快捷筛选
- 风险等级快捷筛选

第二层：

- 关键字搜索
- 产品状态
- 产品类型
- 风险等级
- 查询
- 重置

设计原则：

- 这是工具栏，不是主内容区
- 风格要紧凑、连续、低打扰
- 当前筛选状态需要可见

## 5.6 结果摘要区

职责：

- 在表格前清晰说明当前结果是什么
- 减少用户反复回看筛选条件

展示内容：

- 当前页条数
- 总条数
- 当前状态筛选
- 当前关键词
- 一键清空筛选入口

设计原则：

- 轻量、紧凑
- 用于承接工具带与表格

## 5.7 产品列表区

职责：

- 作为任务处理后的主要结果承载区
- 保持高信息密度，但不丢失可扫读性

产品列建议结构：

- 第一行：产品名 + 状态标签
- 第二行：产品类型 / 风险等级 / 更新时间
- 第三行：功能标签
- 第四行：异常信息或驳回摘要

操作列分级：

- 一级：编辑 / 查看
- 二级：提交审核 / 撤回 / 下架
- 三级：审核记录 / 复制 ID / 驳回意见

原则：

- 主操作直出
- 次级操作控制数量
- 辅助操作可收纳到更多菜单

## 5.8 批量操作区

职责：

- 在多选后明确提示用户进入批量处理状态

展示内容：

- 已选择数量
- 当前可执行动作
- 不可执行原因说明

设计原则：

- 不是普通提示条，而是工作状态条
- 需要比当前实现更有“操作上下文”感

## 5.9 审核记录区

职责：

- 查看审核历史和驳回意见

本轮策略：

- 保留现有弹层逻辑
- 下一轮可升级为抽屉式侧边区，以保留上下文

## 6. 视觉风格

整体气质：

- 专业
- 克制
- 金融信任感强
- 数据清晰

视觉规则：

- 页面背景使用浅灰白
- 普通模块使用白底卡片 + 细边框
- 高优先级模块可使用轻浅品牌蓝底
- 状态色清晰，但避免过饱和
- 减少大面积强渐变

颜色语义：

- 主品牌：深蓝
- 高价值点缀：金色
- 成功 / 已上架：绿色
- 待审核 / 信息状态：蓝色
- 驳回 / 风险：红色

## 7. 组件改动建议

推荐拆分或重构的页面块：

- 顶部工作区
- 任务卡区
- 优先处理区
- 筛选工具带
- 结果摘要区
- 批量操作条

推荐新增或演进组件：

- `ProductWorkbenchHero`
- `ProductTaskCards`
- `ProductUrgentList`
- `ProductFilterToolbar`
- `ProductResultSummary`
- `ProductBatchBar`

第一版允许先不拆文件，但应按上述区块重构页面结构。

## 8. 实施范围

本轮可以直接调整的内容：

- 页面结构顺序
- 模块权重
- 视觉层级
- 表格内容呈现方式
- 操作优先级
- 批量操作条表现

本轮不调整的内容：

- API 数据结构
- 查询参数
- 分页逻辑
- 批量处理逻辑
- 审核记录数据获取逻辑

## 9. 实施优先级

建议按如下顺序推进：

1. 重构首屏工作区
2. 强化任务卡与优先处理区
3. 合并筛选区为统一工具带
4. 增加结果摘要区
5. 优化表格产品列与操作列
6. 增强批量操作条

## 10. 成功标准

完成后页面应达到：

- 打开页面后优先看到任务，而不是先看到表格
- 页面看起来像投顾运营工作台，而不是普通后台列表页
- 筛选与结果形成连续工作流
- 行内信息更易扫读
- 批量处理状态更加明确
- 整体更有金融专业感和秩序感
